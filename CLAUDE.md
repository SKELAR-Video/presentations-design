# SKELAR Presentations Design — правила проєкту

## Інваріанти (нульова толерантність)

> Нові узгоджені правила → одразу сюди, щоб не губились після стиснення контексту.

0. **Текст не обрізати** — ніколи не додавати «…» і не вкорочувати контент мовчки. Якщо текст не влазить — це FAIL валідатора; виправляти треба в промпті LLM, а не в коді обробки.
1. **Геометрія / кеглі / кольори** — не міняти довільно. Усі числа беруться з майстра або з цього файлу.
2. **Текст не вилазить за межі слайда** — кожен елемент повністю в 0–1920×0–1080 Figma px (0–720×0–405 pt).
3. **Нема накладань** — body-текст не перекриває картки; між ними відступ ≥ `gap_min` (30 px).
4. **Одна тема на дек** — або `dark`, або `red`; не змішувати.
5. **Бейдж SKELAR** — завжди присутній біля (1730, 100), не зсувати.
6. **autofitType = NONE** — в усіх TEXT_BOX без винятків.
7. **kpi_cards** — `КАРТКА_N_ЗНАЧЕННЯ` містить тільки числові значення; список або речення → body, не в картку.
8. **НЕ казати «готово»**, поки `validation.pass === true`. FAIL → виправляй сам, не перекидай на користувача.
9. **Нуль втрати контенту** — КОЖЕН ненульовий текстовий слот із плану має зʼявитись у деку. Якщо слот є у плані, але відсутній у `compositions.ts` — це FAIL `content_integrity`. Нові слоти → спочатку додати до композиції та майстра, потім генерувати. Ніколи не ігнорувати текст мовчки.
10. **LLM не генерує прозу — тільки маппінг** — `mapToPlan` (і `mapSlides1to1`) повертають ІНДЕКСИ фрагментів; builder вставляє дослівний текст із входу. LLM не пише й не перефразовує слоти. Валідатор `content_integrity` перевіряє, що кожен рядок кожного слота є точним підрядком вхідного ТЗ. Скорочення НІКОЛИ не відбувається в коді — якщо текст перевищує `max_chars`, це FAIL валідатора; правити треба в самому ТЗ або в промпті LLM, а не в коді.

## Авто-пуш (`lib/auto-push.ts`)

- **Пуш тільки після PASS**. `autoPushIfPass(validation, msg)` — нічого не пушить, якщо `pass === false`.
- **.env.local захищений**: `.gitignore` має `.env*` — перевірка вбудована у функцію; без неї пуш блокується.
- **Пуш у поточну гілку** (`git branch --show-current`).
- На Vercel (`process.env.VERCEL`) — завжди skip.

## Заборони
- **НІКОЛИ не читати `.env.local`** — там ключі, користувач вставляє їх сам
- Не ділитись ключами в чаті

## Бренд — типографіка
- Шрифт: **Inter, weight 500** скрізь і завжди
- `bold: false` — абсолютне правило, без винятків (навіть для заголовків)
- Ієрархія — тільки через розмір шрифту, не через жирність
- Кроки розміру **баді-тексту**: 14, 18, 22pt (інші не додавати без узгодження)

## Бренд — кольори
| Змінна  | HEX       | Призначення                          |
|---------|-----------|--------------------------------------|
| DARK    | `#090D17` | фон слайду (основний)                |
| CARD    | `#1A1F2E` | фон бенто-карток                     |
| RED     | `#FD3433` | фон акцентних слайдів (section_red)  |
| WHITE   | `#FFFFFF` | заголовки, значення KPI              |

Баді-текст завжди: MUTED на DARK, PINK на RED.

## Координатна система
- Референс: **1920×1080 Figma px**
- Google Slides = 9 144 000 × 5 143 500 EMU
- `FPX = 9144000 / 1920 = 4762.5` EMU на Figma-піксель
- Конвертація шрифту: `pt × 2.667 = Figma px`
- **Ніколи не використовувати PX=9525** (це 96dpi/960×540 — неправильна шкала)

## Сітка слайду (всі значення в Figma px)
```
W=1920, H=1080
PAD=100     — мертва зона з усіх боків
UW=1720     — W - 2×PAD, робоча ширина
TH=100      — висота рядка заголовку
TG=100      — відступ між заголовком і бенто-картками
GAP=30      — відступ між бенто-картками
INN=30      — внутрішній відступ усередині картки з усіх 4 сторін (однаковий)
CY=300      — PAD + TH + TG, верхній край карток
CH=680      — H - PAD - CY, висота карток
```

## Логотип
- Розмір: 90×90 px
- Позиція: правий верхній кут у межах сітки — `x = W - PAD - 90 = 1730, y = PAD = 100`
- URL: `process.env.LOGO_URL` — якщо не заданий, fallback = raw GitHub URL
- Raw GitHub URL (вшитий як дефолт): `https://raw.githubusercontent.com/SKELAR-Video/presentations-design/main/public/assets/SKELAR%20Symbol.png`
- Файл лежить: `public/assets/SKELAR Symbol.png`

## Бенто-картки — скруглення кутів
- Google Slides REST API **не дозволяє** задати фіксований радіус для `ROUND_RECTANGLE`
- Використовуємо симуляцію: `RECTANGLE` + 4 кути × (квадрат кольору фону + еліпс кольору картки)
- Радіус: `R=30px`, фіксований незалежно від розміру картки

## Right Bento — константи (Figma виміряно)
```
RBW = 860   — ширина bento-колонки праворуч
RBH = 880   — висота (H - 2×PAD)
RBX = 960   — PAD + LTW + GAP
LTW = 830   — ширина лівої текстової зони (UW - RBW - GAP)
```
Ліва зона: ЗАГОЛОВОК (44pt, 830×260) + ТЕКСТ (22pt, 830×590) — починаючи від PAD.

## Бенто-картки — висота за вмістом (НОВА ЛОГІКА)

**Правило**: `cardH = contentH + 2 × VERT_PAD`. Картки НЕ розтягуються на весь слайд.

- `contentH` — оцінена висота тексту при обраному pt
- `VERT_PAD = 40px` для бенто-рядів (`two_columns`, `three_columns`, `bento_right_*`)
- Усі картки ряду — однакової висоти (= найвища з них)
- Ряд центрується у контент-зоні: `rowY = CY + max(0, (680 - cardH) / 2)`
- Зайвий простір — **зовні карток** (зверху/знизу ряду), НЕ всередині

Реалізація: `buildBentoRowLayoutRequests()` у `lib/google.ts` — переміщує card body + corner squares + corner ellipses + text boxes.

## Бенто-картки — буліти (НОВА ЛОГІКА)

**Правило**: якщо вміст картки — список пунктів → кожен пункт з нового рядка з маркером `•`.

Автоматична конвертація (`preprocessBentoText` у `lib/google.ts`):
- `«пункт1 · пункт2»` → `«• пункт1\n• пункт2»`
- `«рядок1\nрядок2»` (не value+label) → `«• рядок1\n• рядок2»`

**Виняток**: цифра+підпис (`$5M\nнові клієнти`) — не конвертується.

## Бенто-картки — шрифт
- Застосовується до: `two_columns`, `three_columns`, `bento_right_2`, `bento_right_3`, `bento_right_2x2`
- **Старт зі значення role max** (не 18pt), зменшення лише якщо найдовший текст не влазить
- Role max: `two_columns`→48pt, `three_columns`→28pt, `bento_right_2`→36pt, `bento_right_3`→22pt, `bento_right_2x2`→22pt
- Єдиний pt для всього ряду (за найдовшим текстом)
- `updateTextStyle` виконується **після** `replaceAllText` (не до)

## Кольорування тексту з ":"
- Правило діє на **всі не-заголовкові текстові слоти** (body, caption, підзаголовки) — у всіх композиціях
- Якщо текст містить `:` — частина до і включно з `:` стає **WHITE**
- Частина після `:` залишається MUTED (або PINK на червоному тлі)
- ЗАГОЛОВОК — завжди виключений

## Workflow при змінах макету
1. Змінити `create-master/route.ts`
2. Якщо змінились константи сітки (`PAD`, `INN`, `TG` тощо) — оновити ті ж константи в `lib/google.ts` (блок `_PAD`, `_INN` тощо)
3. Зайти на `/setup`, створити новий шаблон
4. Скопіювати новий `MASTER_DECK_ID` в `.env.local`
5. Перезапустити сервер

## Пост-генераційний валідатор (`lib/validator.ts`)

`buildPresentation` повертає `{ url, validation }`.  
`validateDeck(slidesApi, presentationId, plan, planPageIds)` читає фінальний дек через Slides API і перевіряє:

| # | ID перевірки | Що перевіряємо |
|---|---|---|
| 1 | `bounds` | Кожен елемент повністю в межах 0–1920×0–1080 Figma px |
| 2 | `autofit_none` | autofitType усіх TEXT_BOX = NONE |
| 3 | `font_inter_medium` | Шрифт Inter, bold=false у всіх textRun |
| 4 | `max_chars` | Довжина тексту кожного слоту ≤ max_chars зі `compositions.ts` |
| 5 | `skelar_badge` | Елемент-логотип присутній біля (1730, 100) ±20px |
| 6 | `kpi_numeric_values` | Слоти КАРТКА_N_ЗНАЧЕННЯ — тільки числові значення (цифри/+/%) |
| 7 | `kpi_gap` | body-текст не перетинає ряд карток: gap ≥ gap_min (30px) |
| 8 | `theme_consistency` | Тема одна на весь дек (dark або red) |

**Жорстке правило для Claude:**  
НЕ оголошувати «готово», поки `validation.pass === true`.  
Якщо хоч один FAIL — аналізуй причину, правь **сам**, без перекидання на користувача.  
Коли все PASS — дати лінк + вивести звіт (PASS/FAIL по слайдах).

### kpi_cards — правила контенту
- КАРТКА_N_ЗНАЧЕННЯ: ВИКЛЮЧНО числові значення (наприклад: `+42%`, `$5M`, `×2`, `< 0.5%`)
- Речення або список у картці = FAIL (переносити в ТЕКСТ або body)
- Кількість карток = кількість реальних метрик; порожні картки = FAIL
- ТЕКСТ у kpi_cards ≤ 70 символів (короткий субтитул, НЕ абзац)

### Адаптивна висота kpi_cards (ОНОВЛЕНО)
- `computeKpiAdaptive()` у `lib/google.ts` — розраховує висоту карток і тіла
- **Висота за вмістом**: `cardH = valH + lblH + 2×INN + 2×KPI_VERT_PAD` (не заповнює слайд знизу)
- `KPI_VERT_PAD = 30px` — відступ зверху (над цифрою) і знизу (під підписом)
- **Tight group**: ЗНАЧЕННЯ одразу над ПІДПИСОМ (`boxY_label = kCY + INN + KPI_VERT_PAD + valH`)
- Ряд центрується у доступній зоні нижче заголовка/body
- Шрифт ЗНАЧЕННЯ: scale [48, 36, 28, 22, 18, 14] — найбільший, при якому текст не виходить за 3 рядки

### kpi_cards — геометрія майстра (в Figma px)
```
kCY0 = 356  = PAD + TH + subH + TG (оригінальна позиція карток)
kCH0 = 624  = H - PAD - kCY0 (оригінальна висота карток)
kw   = 407  = (UW - 3×GAP) / 4
R    = 30   — радіус заокруглення картки
```
