# SKELAR Presentations Design — правила проєкту

## Інваріанти (нульова толерантність)

> Нові узгоджені правила → одразу сюди, щоб не губились після стиснення контексту.

0. **Текст не обрізати** — ніколи не додавати «…» і не вкорочувати контент мовчки. Якщо текст не влазить — це FAIL валідатора; виправляти треба в промпті LLM, а не в коді обробки.
1. **Геометрія / кеглі / кольори** — не міняти довільно. Усі числа беруться з майстра або з цього файлу.
2. **Текст не вилазить за межі слайда** — кожен елемент повністю в 0–1920×0–1080 Figma px (0–720×0–405 pt).
3. **Нема накладань** — body-текст не перекриває картки; між ними відступ ≥ `gap_min` (30 px).
4. **Одна тема на дек** — або `dark`, або `red`; не змішувати.
5. **Бейдж SKELAR** — завжди присутній біля (1730, 100), не зсувати.
6. **autofitType = NONE** — в усіх TEXT_BOX без винятків.
7. **kpi_cards** — `КАРТКА_N_ЗНАЧЕННЯ` містить тільки числові значення; список або речення → body, не в картку.
8. **НЕ казати «готово»**, поки `validation.pass === true`. FAIL → виправляй сам, не перекидай на користувача.
9. **Нуль втрати контенту** — КОЖЕН ненульовий текстовий слот із плану має зʼявитись у деку. Якщо слот є у плані, але відсутній у `compositions.ts` — це FAIL `content_integrity`. Нові слоти → спочатку додати до композиції та майстра, потім генерувати. Ніколи не ігнорувати текст мовчки.
10. **LLM не генерує прозу — тільки маппінг** — `mapToPlan` (і `mapSlides1to1`) повертають ІНДЕКСИ фрагментів; builder вставляє дослівний текст із входу. LLM не пише й не перефразовує слоти. Валідатор `content_integrity` перевіряє, що кожен рядок кожного слота є точним підрядком вхідного ТЗ. Скорочення НІКОЛИ не відбувається в коді — якщо текст перевищує `max_chars`, це FAIL валідатора; правити треба в самому ТЗ або в промпті LLM, а не в коді.

## Авто-пуш (`lib/auto-push.ts`)

- **Пуш тільки після PASS**. `autoPushIfPass(validation, msg)` — нічого не пушить, якщо `pass === false`.
- **.env.local захищений**: `.gitignore` має `.env*` — перевірка вбудована у функцію; без неї пуш блокується.
- **Пуш у поточну гілку** (`git branch --show-current`).
- На Vercel (`process.env.VERCEL`) — завжди skip.

## Заборони
- **НІКОЛИ не читати `.env.local`** — там ключі, користувач вставляє їх сам
- Не ділитись ключами в чаті

## Бренд — типографіка
- Шрифт: **Inter, weight 500** скрізь і завжди
- `bold: false` — абсолютне правило, без винятків (навіть для заголовків)
- Ієрархія — тільки через розмір шрифту, не через жирність
- Кроки розміру **баді-тексту**: 14, 18, 22pt (інші не додавати без узгодження)
- **Нерозривний пробіл після коротких слів** — літерні слова 1–4 символи (у, і, та, на, до, від, як, це…) не висять у кінці рядка: `addNbsp()` замінює пробіл після них на U+00A0. Числові токени — не чіпати. `content_integrity` вважає U+00A0 ≡ пробілу.
- **Крапка наприкінці ЗАГОЛОВОК та бенто-карток** — прибрати при рендері (`stripTrailingPeriod()`). `?`, `!`, `…`, `...` — залишати. Слот `ЗАГОЛОВОК` + всі `КАРТКА_N` / `КОЛОНКА_N`. `content_integrity` проходить автоматично (stripped ⊂ original).

## Координатна система
- Референс: **1920×1080 Figma px**
- Google Slides = 9 144 000 × 5 143 500 EMU
- `FPX = 9144000 / 1920 = 4762.5` EMU на Figma-піксель
- Конвертація шрифту: `pt × 2.667 = Figma px`
- **Ніколи не використовувати PX=9525** (це 96dpi/960×540 — неправильна шкала)

## Сітка слайду (всі значення в Figma px)
```
W=1920, H=1080
PAD=100     — мертва зона з усіх боків
UW=1720     — W - 2×PAD, робоча ширина
TH=100      — висота рядка заголовку
TG=100      — відступ між заголовком і бенто-картками
GAP=30      — відступ між бенто-картками
INN=30      — внутрішній відступ усередині картки з усіх 4 сторін (однаковий)
CY=300      — PAD + TH + TG, верхній край карток
CH=680      — H - PAD - CY, висота карток
TITLE_W=1610 — LOGO_X - LOGO_GAP - PAD; ширина заголовкових боксів у верхній смузі
             — правий край = 1710, зазор до логотипа = 20px
TITLE_GAP=60 — фіксований відступ між низом ЗАГОЛОВОК і верхом ПІДЗАГОЛОВОК/ТЕКСТ
             — однаковий на ВСІХ композиціях (cover, section, title_body, bento_right_*)
```

## Логотип
- Розмір: 90×90 px
- Позиція: правий верхній кут у межах сітки — `x = W - PAD - 90 = 1730, y = PAD = 100`
- **Зарезервована зона**: x∈[1730,1820], y∈[100,190] — жоден TEXT_BOX не перетинає
- Заголовкові бокси використовують `TITLE_W=1610` (не `UW=1720`): right=1710, зазор=20px
- Три варіанти логотипу:
  - `public/assets/SKELAR Symbol.png` — темний/стандартний (більшість слайдів)
  - `public/assets/SKELAR Symbol for red.png` — білий (для `section` з червоним фоном)
  - `public/assets/SKELAR Logo.png` — повний wordmark (горизонтальний, для `cover_title_only`)
- URL-пріоритет: `LOGO_URL` / `LOGO_RED_URL` / `LOGO_WORDMARK_URL` env → `VERCEL_URL` → GitHub raw
  - `LOGO_WORDMARK_URL`: якщо не задано — виводиться з `LOGO_URL` (замінює ім'я файлу)
- **Важливо**: файли логотипів мають бути в гіті — Vercel деплоїть тільки те, що є в репо. GitHub raw не працює для приватного репо.
- Логотипи додаються двома окремими `batchUpdate` після основного: `symbolRequests` і `wordmarkRequests` — помилка одного не вбиває інший.

## Фони слайдів

### Cover — випадковий фотофон
- `cover` і `cover_title_only` отримують випадковий фон з `public/assets/backgrounds/` (Mountain 0–5)
- Встановлюється через `updatePageProperties` → `pageBackgroundFill.stretchedPictureFill`
- URL-пріоритет: `BG_BASE_URL` env → `VERCEL_URL` → GitHub raw (приватне репо = GitHub не працює)
- Функція: `getBgBaseUrl()` + `randomCoverBg()` у `lib/google.ts`

### Cover title only — великий заголовок по центру
- `cover_title_only` — коли на першому слайді лише заголовок (без підзаголовка і дати з документа)
- **Макет**: SKELAR Logo.png (wordmark) праворуч вгорі `(1463, 99, 357×90)`; дата-пілюля ліворуч вгорі `(100, 99, 195×115, #292D39)`; ЗАГОЛОВОК по центру кадру `(100, 100, 1720×880)`
- **Дата**: автоматично — поточна дата у форматі `дд.мм.рррр`, генерується `formatCurrentDate()` у `lib/google.ts`
- **Шрифт ЗАГОЛОВОК**: динамічний, кроки `[66, 54, 44, 36, 28, 22]pt`, `pickCoverTitleOnlyPt()`, вирівнювання CENTER+MIDDLE
- **Дата-пілюля**: `ROUND_RECTANGLE` заливка `#292D39`, Inter 500 18pt MUTED, CENTER+MIDDLE — створюється динамічно, не в майстрі
- **Валідатор**: `skelar_badge` → `(1463, 99)`; `logo_overlap` → skip (full-slide title навмисно перекриває зону логотипа)

### Section — червоний фон + білий логотип
- Усі слайди `composition === 'section'` автоматично отримують фон `#FD3433`
- Встановлюється через `updatePageProperties` → `pageBackgroundFill.solidFill`
- Логотип на цих слайдах: `SKELAR Symbol for red.png` (білий варіант)
- `section_red` і `closing` — не зачіпаються цією логікою

### Section/section_red/closing — динамічний шрифт без підзаголовка
- Коли `ПІДЗАГОЛОВОК` відсутній: `buildSectionFloatRequests()` підбирає розмір шрифту через `pickSectionTitlePt()`, кроки `[66, 54, 44, 36, 28, 22]pt`, ширина боксу `TITLE_W=1610`, доступна висота 880px
- Висота боксу ЗАГОЛОВОК — динамічна: `ceil(estimateLineCount × lineH(pt))`, не фіксована 260px
- Коли `ПІДЗАГОЛОВОК` є: все без змін (44pt, фіксована висота 260px, позиція підзаголовка y=420)
- `updateTextStyle` додається тільки якщо обраний pt ≠ 44 (майстер вже має 44pt)

## Бенто-картки — скруглення кутів
- Google Slides REST API **не дозволяє** задати фіксований радіус для `ROUND_RECTANGLE`
- Використовуємо симуляцію: `RECTANGLE` + 4 кути × (квадрат кольору фону + еліпс кольору картки)
- Радіус: `R=30px`, фіксований незалежно від розміру картки

## Right Bento — константи (Figma виміряно)
```
RBW = 860   — ширина bento-колонки праворуч
RBH = 880   — висота (H - 2×PAD)
RBX = 960   — PAD + LTW + GAP
LTW = 830   — ширина лівої текстової зони (UW - RBW - GAP)
```
Ліва зона: ЗАГОЛОВОК (44pt, 830×dynamic) + ТЕКСТ (22pt, 830×dynamic) — ТЕКСТ плаває на TITLE_GAP=60px нижче фактичного низу ЗАГОЛОВОК.

**Порожній ТЕКСТ-бокс**: навіть якщо слот відсутній — бокс переміщується нижче ЗАГОЛОВОК (h=1, невидимий). Без цього майстер-бокс залишається на y=390 і перетинає великий заголовок → `bento_left_overlap` FAIL. Те саме правило в `buildSectionFloatRequests` і `buildTitleBodyFloatRequests`.

## Бенто-картки — висота за вмістом (НОВА ЛОГІКА)

**Правило**: `cardH = contentH + 2 × VERT_PAD`. Картки НЕ розтягуються на весь слайд.

- `contentH` — оцінена висота тексту при обраному pt
- `VERT_PAD = 40px` для бенто-рядів (`two_columns`, `three_columns`, `bento_right_*`)
- Усі картки ряду — однакової висоти (= найвища з них)
- Ряд центрується у контент-зоні: `rowY = CY + max(0, (680 - cardH) / 2)`
- Зайвий простір — **зовні карток** (зверху/знизу ряду), НЕ всередині

Реалізація: `buildBentoRowLayoutRequests()` у `lib/google.ts` — переміщує card body + corner squares + corner ellipses + text boxes.

## Бенто-картки — буліти (НОВА ЛОГІКА)

**Правило**: якщо вміст картки — список пунктів → кожен пункт з нового рядка з маркером `•`.

Автоматична конвертація (`preprocessBentoText` у `lib/google.ts`):
- `«пункт1 · пункт2»` → `«• пункт1\n• пункт2»`
- `«рядок1\nрядок2»` (не value+label) → `«• рядок1\n• рядок2»`

**Виняток**: цифра+підпис (`$5M\nнові клієнти`) — не конвертується.

## Бенто-картки — шрифт
- Застосовується до: `two_columns`, `three_columns`, `bento_right_2`, `bento_right_3`, `bento_right_2x2`
- **Старт зі значення role max** (не 18pt), зменшення лише якщо найдовший текст не влазить
- Role max: `two_columns`→48pt, `three_columns`→28pt, `bento_right_2`→36pt, `bento_right_3`→22pt, `bento_right_2x2`→22pt
- **Єдиний pt для групи**: по кожній картці знаходимо максимальний pt, що влазить; фінальний групповий = мінімум із них (найтісніший текст диктує розмір); застосовується однаковий pt до всіх карток групи
- `updateTextStyle` виконується **після** `replaceAllText` (не до)
- **Safety factor = 1.1** у `textFits`, `textFitsParagraphs`, `pickTitlePt` — (було 1.2; зменшено для кращого використання ширини)
- **Пунктуація стрипується** з країв слів перед вимірюванням ширини в `longestWordPx`: "активність," рахується як 10 символів, не 11 — інакше кома тягнула б групу до меншого pt

## Кольорування тексту з ":"
- Правило діє на **всі не-заголовкові текстові слоти** (body, caption, підзаголовки) — у всіх композиціях
- Якщо текст містить `:` — частина до і включно з `:` стає **WHITE**
- Частина після `:` залишається MUTED (або PINK на червоному тлі)
- ЗАГОЛОВОК — завжди виключений

## Флоу застосунку

```
/ (page.tsx)
  ↓ POST /api/fetch-doc      — завантажує контент Google Doc або Slides
  ↓ POST /api/map            — LLM будує план слайдів
  ↓ POST /api/generate       — копіює майстер, заповнює токени, валідує
  ↓ /result                  — посилання на готовий дек + факти з файлу
```

Проміжного кроку редагування плану **немає** — генерація відбувається одразу після маппінгу.

Сторінки `/plan` і `/preview` залишаються в коді, але не задіяні в основному флоу.

## Workflow при змінах макету
1. Змінити `create-master/route.ts`
2. Якщо змінились константи сітки (`PAD`, `INN`, `TG` тощо) — оновити ті ж константи в `lib/google.ts` (блок `_PAD`, `_INN` тощо)
3. Зайти на `/setup`, створити новий шаблон
4. Скопіювати новий `MASTER_DECK_ID` в `.env.local`
5. Перезапустити сервер

## Пост-генераційний валідатор (`lib/validator.ts`)

`buildPresentation` повертає `{ url, validation }`.  
`validateDeck(slidesApi, presentationId, plan, planPageIds)` читає фінальний дек через Slides API і перевіряє:

| # | ID перевірки | Що перевіряємо |
|---|---|---|
| 1 | `bounds` | Кожен елемент повністю в межах 0–1920×0–1080 Figma px |
| 2 | `autofit_none` | autofitType усіх TEXT_BOX = NONE |
| 3 | `font_inter_medium` | Шрифт Inter, bold=false у всіх textRun |
| 4 | `max_chars` | Довжина тексту кожного слоту ≤ max_chars зі `compositions.ts` |
| 5 | `skelar_badge` | Елемент-логотип присутній біля (1730, 100) ±20px |
| 6 | `kpi_numeric_values` | Слоти КАРТКА_N_ЗНАЧЕННЯ — тільки числові значення (цифри/+/%) |
| 7 | `kpi_gap` | body-текст не перетинає ряд карток: gap ≥ gap_min (30px) |
| 8 | `theme_consistency` | Тема одна на весь дек (dark або red) |
| 9 | `logo_overlap` | Жоден TEXT_BOX не перетинається із зоною логотипа (top-right або bottom-left) |
| 10 | `bento_left_overlap` | ЗАГОЛОВОК і ТЕКСТ в лівій колонці bento_right_* не перекриваються |
| 11 | `bento_trailing_period` | Текст бенто-картки не закінчується одиничною крапкою (після auto-strip) |

**Жорстке правило для Claude:**  
НЕ оголошувати «готово», поки `validation.pass === true`.  
Якщо хоч один FAIL — аналізуй причину, правь **сам**, без перекидання на користувача.  
Коли все PASS — дати лінк + вивести звіт (PASS/FAIL по слайдах).

### kpi_cards — правила контенту
- КАРТКА_N_ЗНАЧЕННЯ: ВИКЛЮЧНО числові значення (наприклад: `+42%`, `$5M`, `×2`, `< 0.5%`)
- Речення або список у картці = FAIL (переносити в ТЕКСТ або body)
- Кількість карток = кількість реальних метрик; порожні картки = FAIL
- ТЕКСТ у kpi_cards ≤ 70 символів (короткий субтитул, НЕ абзац)

### kpi_cards — компактування чисел (КАРТКА_N_ЗНАЧЕННЯ)
Застосовується **тільки** до `КАРТКА_N_ЗНАЧЕННЯ` (Step 2.60 у `lib/google.ts`). Звичайний текст-абзац — не чіпати.

Алгоритм: прибрати пробіли-розділювачі → порахувати цифри.

| Цифр | Умова | Дія | Приклади |
|------|-------|-----|---------|
| ≤ 4 | завжди | без змін | `1500`→`1500`; `2026`→`2026` |
| 5–6 | `value % 100 = 0` | → K | `10 000`→`10K`; `12 500`→`12.5K` |
| 5–6 | `value % 100 ≠ 0` | без змін | `12 345`→`12 345` |
| 7+  | `value % 100 000 = 0` | → M | `2 000 000`→`2M`; `2 500 000`→`2.5M` |
| 7+  | `value % 100 000 ≠ 0` | без змін | `2 456 789`→`2 456 789` |

- Максимум 1 знак після коми; `2.0M`→`2M` (прибирати `.0`)
- Префікси/суфікси зберігаються: `$2 000 000`→`$2M`; `10 000+`→`10K+`
- `«2M»` ↔ `«2 000 000»` = PASS у `content_integrity` (валідатор: `isCompactNumberMatch`)
- Неокруглі числа лишаються дослівно — кожен розряд важливий

### Адаптивна висота kpi_cards (ОНОВЛЕНО)
- `computeKpiAdaptive()` у `lib/google.ts` — розраховує висоту карток і тіла
- **Висота за вмістом**: `cardH = valH + lblH + 2×INN + 2×KPI_VERT_PAD` (не заповнює слайд знизу)
- `KPI_VERT_PAD = 30px` — відступ зверху (над цифрою) і знизу (під підписом)
- **Tight group**: ЗНАЧЕННЯ одразу над ПІДПИСОМ (`boxY_label = kCY + INN + KPI_VERT_PAD + valH`)
- Ряд центрується у доступній зоні нижче заголовка/body
- Шрифт ЗНАЧЕННЯ: scale [48, 36, 28, 22, 18, 14] — найбільший, при якому текст не виходить за 3 рядки

## Робоче правило: перевірений цикл (обов'язкове)

ГОЛОВНЕ: "готово" визначається ФАКТАМИ з `/api/inspect-deck` (реальний файл), НІКОЛИ словом чи математикою. Не звітувати "done" на основі того, що функція порахувала — тільки на основі того, що реально лежить у згенерованому деку.

Розподіл: correctness перевіряє машина через inspect-deck (контент присутній, fontSize, геометрія, кількість слайдів, межі); смак перевіряє людина (краса/читабельність).

Цикл на кожну правку:
1. Одна зміна за раз.
2. Перегенерувати тестовий бриф (10 аркушів).
3. Прочитати новий дек через inspect-deck — факти з файлу.
4. Автоперевірка фактів проти очікуваного → вивести ЧИСЛА + PASS/FAIL по кожному слайду.
5. Червоне → виправити самому (макс 2 спроби) → крок 2. Після 2 невдач — СТОП і звіт числами, без нескінченного циклу.
6. Зелене → повідомити людині ЧИСЛА (не "готово"). Людина дивиться на смак.
7. Дефект, який впіймала людина, стає новою автоперевіркою (щоб не вертався).

Заборонено: звітувати "готово" за математикою/наміром коду; зелений PASS без фактів із файлу; нескінченні спроби "майже полагодив".
