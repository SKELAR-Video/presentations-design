# SKELAR Presentations Design — правила проєкту

## Огляд проєкту

**Тулза**: Next.js App Router — генерує Google Slides через REST API з Google Docs/Slides брифу.

**Стек**: Next.js 14+, TypeScript, Google Slides API v1, Anthropic Claude, Vercel.

**Флоу**:
```
/ (page.tsx)
  ↓ POST /api/fetch-doc      — завантажує Google Doc або Slides
  ↓ POST /api/map            — LLM будує план слайдів (тільки маппінг, не текст)
  ↓ POST /api/generate       — копіює майстер, заповнює токени, валідує
  ↓ /result                  — посилання на готовий дек + факти з файлу
```
Проміжного кроку редагування плану **немає**. Сторінки `/plan` і `/preview` — в коді, але не в активному флоу.

**Ключові файли**:
| Файл | Призначення |
|------|-------------|
| `app/page.tsx` | Input → fetch-doc → map → generate → `/result` |
| `app/result/page.tsx` | Посилання на дек + панель фактів |
| `lib/google.ts` | Генерація дека (копія шаблону + заміна токенів + геометрія) |
| `lib/compositions.ts` | Список та опис усіх композицій |
| `lib/anthropic.ts` | Маппінг брифу → план слайдів через LLM |
| `lib/validator.ts` | Пост-генераційна валідація дека |
| `app/api/create-master/route.ts` | Створення майстер-шаблону Google Slides |

**Workflow при змінах макету**: змінити `create-master/route.ts` → оновити ті ж константи в `lib/google.ts` → `/setup` → новий `MASTER_DECK_ID` → `.env.local` → перезапустити сервер. Зміни в `lib/google.ts` (кегль, геометрія) **не потребують rebuild master** — вступають в силу з наступної генерації.

## Безпека

- **НІКОЛИ не читати `.env.local`** — там ключі, користувач вставляє їх сам
- **Не ділитись ключами в чаті**

## Стиль відповідей

- Лаконічно в тексті: без преамбул, без повторів питання, без фінальних сумарі
- Показуй код/числа, а не пояснення про пояснення
- Повна дисципліна в роботі: root-cause перед фіксом, одна зміна за раз, перевірений цикл

## SKELAR бренд

- Шрифт Inter, weight 500 — **ніколи bold**
- Ієрархія тільки через розмір шрифту

## Перед публічним шером тулзи

Попросити helpdesk/адміна skelar.tech налаштувати **Domain-Wide Delegation (DWD)**:
- Мета: прибрати `drive` scope з юзер-токена (зараз застосунок запитує повний доступ до Drive)
- Що робить адмін: Google Cloud Console → Service Account → Enable DWD → Google Workspace Admin → Security → API Controls → додати client ID сервіс-акаунту зі scopes `drive`, `presentations`
- Час: ~15 хвилин

## Робоче правило: перевірений цикл (обов'язкове)

ГОЛОВНЕ: "готово" визначається ФАКТАМИ з `/api/inspect-deck` (реальний файл), НІКОЛИ словом чи математикою. Не звітувати "done" на основі того, що функція порахувала — тільки на основі того, що реально лежить у згенерованому деку.

Розподіл: **машина** перевіряє correctness (контент, fontSize, геометрія, кількість слайдів, межі); **людина** перевіряє смак (краса/читабельність).

Цикл на кожну правку:
1. Одна зміна за раз.
2. Перегенерувати тестовий бриф (10 аркушів).
3. Прочитати новий дек через inspect-deck — факти з файлу.
4. Автоперевірка фактів проти очікуваного → вивести ЧИСЛА + PASS/FAIL по кожному слайду.
5. Червоне → виправити самому (макс 2 спроби) → крок 2. Після 2 невдач — СТОП і звіт числами, без нескінченного циклу.
6. Зелене → повідомити людині ЧИСЛА (не "готово"). Людина дивиться на смак.
7. Дефект, який впіймала людина, стає новою автоперевіркою (щоб не вертався).

Заборонено: звітувати "готово" за математикою/наміром коду; зелений PASS без фактів із файлу; нескінченні спроби "майже полагодив".

## Діагноз перед фіксом: bug чи design gap

Перш ніж правити код на FAIL — визнач тип проблеми:

- **Content/code bug** — текст є у вхідному ТЗ, слот існує в композиції, але щось у коді (маппінг, ключі, кодування) губить зв'язок. Це чинити кодом.
- **Design gap (capacity)** — контенту більше елементів, ніж slots у master-шаблоні композиції (напр. 4 колонки тексту, а `three_columns`/`bento_right_3`/`three_columns_num` мають рівно 3 картки в майстрі). Це **не баг коду** — рендер фізично не має куди покласти 4-й елемент. Жоден guard/rename/normalize у валідаторі це не виправить.

Ознака design gap: одна й та сама помилка (`unmapped`, `content lost`) стабільно повторюється на певному N елементів для композицій з фіксованою ємністю, а композиція з динамічною геометрією (`columns_flex`) на тому ж контенті проходить PASS.

Якщо це design gap — **не намагатись фіксити код** (макс 1 спроба на перевірку гіпотези). Замість цього сказати людині прямо: "нема дизайну на N елементів для composition X, потрібен новий master або динамічна геометрія" — і чекати рішення (новий Figma-дизайн чи перепис композиції на flex), а не еміти 10 спроб патчів.

**Як додавати нове правило:**
- Визнач scope → клади у файл, чий scope підходить.
- Жоден scope не підходить → запропонуй новий файл (назва + scope) і чекай підтвердження; **не створюй самостійно**.
- Не дублюй між файлами.
- Додав новий файл → онови цю таблицю.
